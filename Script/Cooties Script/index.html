Multiplayer Architecture Basics

Understanding Client-Server Model

In Cooties, we use a peer-to-peer client-server architecture.

This means:

  1. One peer acts as the SERVER - It has authority over game logic, manages state, and validates actions
  2. Other peers are CLIENTS - They send input to the server and receive state updates

Why Server Authority Matters

Server authority prevents cheating and ensures consistency.

In our game Cooties, the server:

  • Decides which player to infect when the round begins
  • Tracks scores
  • Manages round timers

Clients only:

  • Send their position (movement, jumps)
  • Synchronize their animation and if the sprite is flipped
  • Receive and display the game state
  • Play animations and sound effects

Here's how we enforce server authority in our game:

res://Scenes/Game/game.gd:27

func _ready() -> void:
	add_to_group("game")

	# Only server manages game logic
	if not multiplayer.is_server():
		return

	Steam.setLobbyJoinable(SteamInit.lobby_id, false)
	_setup_timers()
	multiplayer.peer_disconnected.connect(_on_peer_disconnected)

	await get_tree().create_timer(0.5).timeout
	_start_round()

Notice the early return for clients - they skip all the game logic initialization!

The Three Key Components

[VISUAL: Three-column layout showing the components]

Every multiplayer game in Godot needs:

  1. MultiplayerPeer - Handles the low-level networking
  2. RPC (Remote Procedure Calls) - Sends commands across the network
  3. MultiplayerSpawner/Synchronizer - Manages object creation and state replication

We'll explore each of these in detail throughout this tutorial.

Establishing a connection using Godot's Multiplayer Peer

[VISUAL: Split screen comparing ENET and Steam logos]

What is a MultiplayerPeer?

A MultiplayerPeer is Godot's abstraction for network communication.

It handles:

  • Establishing connections
  • Sending and receiving data packets
  • Managing peer IDs
  • Connection reliability

Godot 4 supports multiple backends, but the two most common are ENET and Steam. But there's also the Nakama Bridge and the Epic Online Services peers

On Cooties' main menu we give the player the choice between selecting the ENET or Steam multiplayer peer with unique UI for both.

ENET MultiplayerPeer

ENET is a lightweight, open-source networking library.

It's perfect for:

  • Quick prototyping
  • Direct IP connections
  • LAN parties
  • Games with dedicated servers

Creating an ENET Server:

Scenes/Games/game.gd:126

func _on_host_game_pressed() -> void:
	match backend.selected:
		MultiplayerBackend.ENET:
			print("Creating ENET Server")
			var peer: ENetMultiplayerPeer = ENetMultiplayerPeer.new()
			var error: Error = peer.create_server(7777, 4)
			if error:
				print("Server Error: %s" % error)
			else:
				multiplayer.multiplayer_peer = peer
				Global.add_local_player()
				Global.change_level("res://Scenes/Lobby/lobby.tscn")

Key points:

  • create_server(7777, 4) - Port 7777, maximum 4 players
  • multiplayer.multiplayer_peer = peer - Tells Godot to use this peer
  • Error handling is essential for networking!

Creating an ENET Client:

func _on_connect_pressed() -> void:
	var peer: ENetMultiplayerPeer = ENetMultiplayerPeer.new()
	var error: Error = peer.create_client(ip_address.text, 7777)
	if error:
		print("Client Error: %s" % error)
	else:
		Global.ip_address = ip_address.text
		multiplayer.multiplayer_peer = peer

		# Wait for player data to sync from server
		await Global.players_synced
		Global.change_level("res://Scenes/Lobby/lobby.tscn")

Notice:

  • Client needs the server's IP address
  • We await for player data to sync before changing scenes
  • This prevents race conditions!

Steam MultiplayerPeer

Steam networking is more robust and feature-rich. Benefits include:

  • NAT punchthrough - Players connect without port forwarding
  • Relay servers - Steam provides backup routing if direct connection fails
  • Security through Steam Sockets - No leaking IPs for DDOS' or potential attacks

Initializing Steam:

extends Node

var steam_running: bool = true
var lobby_id: int = 0
var peer: SteamMultiplayerPeer = SteamMultiplayerPeer.new()
var steam_name: String

func _ready() -> void:
	process_mode = Node.PROCESS_MODE_ALWAYS
	var init_steam: Dictionary = Steam.steamInitEx(480)
	print("Steam Init: %s" % init_steam)

	if init_steam['status'] > Steam.STEAM_API_INIT_RESULT_OK:
		steam_running = false
	else:
		steam_name = Steam.getPersonaName()

func _process(_delta: float) -> void:
	Steam.run_callbacks()

Critical: You must call Steam.run_callbacks() every frame!

Creating a Steam Lobby:

	if connected == 1:

Understanding RPC Calls

[VISUAL: Animation showing function call traveling across network]

Now that we've established our connection, we need to communicate between our peers.

What is an RPC?

RPC stands for Remote Procedure Call.

It's a way to call a function on another computer across the network.

Think of it like this:

  • Normal function call: Executes on your computer
  • RPC call: Executes on other players' computers
# Normal function call - only runs locally
_update_score(10)

# RPC call - runs on all connected peers
_update_score.rpc(10)

RPC Annotations

In Godot, we define RPCs using the @rpc annotation. Let's break down the syntax:

# Only called by the server, also ran locally
@rpc("authority", "call_local", "reliable")
func server_local_function():
	pass

# Called by any peer, only executed on other peers
@rpc("any_peer", "call_remote", "reliable")
func client_remote_function():
        pass

First Parameter: Who Can Call?

  • "authority" - Only the server can call this RPC
  • "any_peer" - Any connected peer can call it

Second Parameter: Where Does It Execute?

  • "call_local" - Runs on the caller AND all remote peers
  • "call_remote" - Only runs on remote peers, NOT the caller

Third Parameter: Reliability

  • "reliable" - Guaranteed delivery (TCP-like)
  • "unreliable" - Best effort, may be dropped (UDP-like)
  • "unreliable_ordered" - May be dropped, but ordered if delivered

RPC Pattern #1: Server Authority Broadcast

[VISUAL: Diagram showing server sending to all clients]

The Server makes a decision and tells everyone.

res://Scenes/Game/game.gd:276

# RPC: Set player infection state
@rpc("authority", "call_local", "reliable")
func _set_player_infected(peer_id: int, infected: bool) -> void:
	var player: Player = players_node.get_node_or_null(str(peer_id))
	if player:
		player.set_infected(infected)

	if hud:
		hud.update_infection_display(peer_id, infected)

Calling it:

# Server decides who to infect
_set_player_infected.rpc(random_infected_peer_id, true)

What happens:

  1. Server calls .rpc() on the function
  2. Godot sends the call to all connected peers
  3. Each peer (including server, due to call_local) executes the function
  4. Everyone sees the infected player update

When to use:

  • Updating game state (scores, health, positions)
  • Triggering events (round start, game over)
  • Visual effects everyone should see

RPC Pattern #2: Client Request to Server

[VISUAL: Diagram showing client sending to server only]

Client asks server for information or to perform an action.

@rpc("any_peer", "call_remote", "reliable")
func get_player_from_server(peer_id: int) -> void:
	if multiplayer.is_server():
		if Global.players.has(peer_id):
			send_player_to_peer.rpc_id(
				multiplayer.get_remote_sender_id(),
				peer_id,
				Global.players[peer_id]
			)

Calling it:

# Client requests player data
get_player_from_server.rpc(peer_id)

What happens:

  1. Client calls .rpc() which sends to server only (call_remote)
  2. Only the server executes the function
  3. Server identifies the caller with get_remote_sender_id()
  4. Server responds with targeted rpc_id() back to that client

When to use:

  • Client needs data from server
  • Client requests an action (joining team, purchasing item)
  • Validation checks (server verifies client can do something)

RPC Pattern #3: Targeted RPC with rpc_id()

[VISUAL: Diagram showing server sending to specific client]

Send an RPC to a specific peer, not everyone.

@rpc("authority", "call_remote", "reliable")
func send_player_to_peer(peer_id: int, player_data: Dictionary) -> void:
	Global.players[peer_id] = player_data
	player_info_updated.emit(peer_id)

Calling it with a target:

# Send to specific peer (not everyone)
send_player_to_peer.rpc_id(target_peer_id, peer_id, Global.players[peer_id])

Key difference: .rpc_id(peer_id, ...) instead of .rpc(...)

When to use:

  • Sending data to specific player (inventory, private messages)
  • Late-joining player needs full state sync
  • Notification meant for one player

RPC Pattern #4: Any Peer Broadcast

[VISUAL: Diagram showing client sending to everyone]

Less common, but useful: Any player can trigger an update for everyone.

# Updates the character sprite texture across all clients
@rpc("any_peer", "call_local", "reliable")
func _update_character_sprite(index: int) -> void:
	var character_texture: Resource

	match index:
		Global.Characters.VIRTUALGUY:
			character_texture = load("res://Assets/Images/Characters/Virtual Guy/Jump (32x32).png")
		Global.Characters.PINKMAN:
			character_texture = load("res://Assets/Images/Characters/Pink Man/Jump (32x32).png")
		Global.Characters.NINJAFROG:
			character_texture = load("res://Assets/Images/Characters/Ninja Frog/Jump (32x32).png")
		Global.Characters.MASKDUDE:
			character_texture = load("res://Assets/Images/Characters/Mask Dude/Jump (32x32).png")

	character_sprite.texture = character_texture

	# Update Global via RPC to sync across all clients
	var peer_id: int = multiplayer.get_remote_sender_id()
	Global.set_player_character.rpc(peer_id, index)

When to use:

  • Cosmetic changes (character selection, emotes)
  • Non-critical state changes
  • Chat messages

Warning: Be careful with any_peer - validate on server to prevent cheating!

RPC Best Practices

[VISUAL: Checklist animation]

  1. Use "reliable" for important state changes - Game logic, scores, positions
  2. Use "unreliable" for frequent cosmetic updates - Particle effects, sounds (not used in this project but useful for high-frequency data)
  3. Always validate on server for any_peer RPCs - Never trust client input
  4. Minimize RPC calls - Send "player moved to position X" not "player pressed W key"
  5. Group related data - Send one RPC with multiple parameters instead of many RPCs

Common RPC Mistakes

[VISUAL: Red X marks over code]

❌ Calling RPC before setting multiplayer_peer:

my_function.rpc()  # This will fail!
multiplayer.multiplayer_peer = peer  # Set peer FIRST

❌ Using call_remote when you want local execution too:

@rpc("authority", "call_remote", "reliable")  # Server won't see the change!
func update_ui(text: String):
	label.text = text

Use call_local if the caller should also execute!

❌ Forgetting server authority checks:

@rpc("any_peer", "call_local", "reliable")
func set_player_score(score: int):
	# Missing: if not multiplayer.is_server(): return
	Global.player_score = score  # Clients could cheat!

MultiplayerSpawner (15:00 - 20:00)

[VISUAL: Animation showing players popping into existence]

What is MultiplayerSpawner?

MultiplayerSpawner is a Godot node that automatically replicates object creation across the network.

Without it, you'd need to:

  1. Spawn object on server
  2. Send RPC to all clients
  3. Clients manually instantiate the same object
  4. Sync the initial state manually

MultiplayerSpawner does all of this automatically!

The PlayerSpawner Script

[VISUAL: Code walkthrough with highlights]

class_name PlayerSpawner
extends MultiplayerSpawner

const PLAYER_SCENE = preload("res://Scenes/Player/player.tscn")

@onready var players: Node = $Players
@onready var spawn_points: Array[Node] = get_node("SpawnPoints").get_children()


func _ready() -> void:
	spawn_function = spawn_player

	multiplayer.peer_disconnected.connect(_on_peer_disconnected)

	# Only the server should spawn players
	if multiplayer.is_server():
		# Spawn all players (including server)
		for peer_id: int in Global.players.keys():
			call_deferred("spawn", peer_id)

Key line: spawn_function = spawn_player

This tells the MultiplayerSpawner to use our custom function instead of simple instantiation.

The Custom Spawn Function

[VISUAL: Step-by-step animation]

func spawn_player(peer_id: int) -> Player:
	var player: Player = PLAYER_SCENE.instantiate()
	var spawn_point: Marker2D = spawn_points.pop_back()

	player.set_multiplayer_authority(peer_id)
	player.name = str(peer_id)
	player.global_position = spawn_point.position

	var character_sprite: SpriteFrames

	match Global.players[peer_id].character:
		Global.Characters.VIRTUALGUY:
			character_sprite = load("res://Scenes/Player/CharacterSprites/virtual_guy.tres")
		Global.Characters.PINKMAN:
			character_sprite = load("res://Scenes/Player/CharacterSprites/pink_man.tres")
		Global.Characters.NINJAFROG:
			character_sprite = load("res://Scenes/Player/CharacterSprites/ninja_frog.tres")
		Global.Characters.MASKDUDE:
			character_sprite = load("res://Scenes/Player/CharacterSprites/mask_dude.tres")

	player.animated_sprite_2d.sprite_frames = character_sprite
	return player

What's happening:

  1. Instantiate - Create player from scene
  2. Get spawn point - Pop a location from the array
  3. Set authority - set_multiplayer_authority(peer_id) makes this peer control the player
  4. Set name - Using peer_id as name makes it easy to find later
  5. Set position - Place at spawn marker
  6. Customize - Load character sprite based on player choice
  7. Return - Give the player back to the spawner

Critical: The function MUST return the instantiated node!

Understanding Multiplayer Authority

[VISUAL: Diagram showing authority ownership]

player.set_multiplayer_authority(peer_id)

This line is crucial! It means:

  • Only this peer can modify the player's input and physics
  • Other peers receive synchronized updates but can't control it
  • Server can override if needed (server always has ultimate authority)

Without this, all players would control every character!

Handling Player Disconnection

func _on_peer_disconnected(peer_id: int) -> void:
	var disconnected_player: Player = players.get_node_or_null(str(peer_id))
	if disconnected_player:
		disconnected_player.queue_free()

When a player leaves:

  1. Find their player node (named by peer_id)
  2. Remove it with queue_free()
  3. MultiplayerSpawner automatically replicates the deletion to all clients

Why Use call_deferred?

[VISUAL: Timeline showing frame execution]

call_deferred("spawn", peer_id)

call_deferred waits until the current frame finishes before spawning.

This prevents issues where:

  • Scene isn't fully loaded yet
  • Other nodes aren't ready
  • Synchronization hasn't initialized

Rule of thumb: Always use call_deferred("spawn", ...) for safety!

Chapter 5: MultiplayerSynchronizer (20:00 - 25:00)

What is MultiplayerSynchronizer?

While MultiplayerSpawner handles creation, MultiplayerSynchronizer handles continuous state updates.

It automatically:

  • Syncs specified properties across the network
  • Sends updates only when values change
  • Interpolates values for smooth visuals
  • Reduces bandwidth by delta compression

How It Works in Practice

[VISUAL: Split-screen showing local player vs remote player]

On the authority peer (the player who owns this character):

func _physics_process(delta: float) -> void:
	# Only process input for the player we control
	if not is_multiplayer_authority():
		return

	# ... movement code ...

	move_and_slide()  # Updates position
	_update_animation()  # Updates animation state

The player:

  1. Processes input locally
  2. Updates position through physics
  3. Updates animation based on movement
  4. MultiplayerSynchronizer automatically sends these changes

On remote peers (everyone else):

func _ready() -> void:
	# ...

	if not is_multiplayer_authority():
		physics_interpolation_mode = Node.PHYSICS_INTERPOLATION_MODE_ON

Remote players:

  1. DON'T process input (early return)
  2. DON'T run physics
  3. ONLY receive synchronized values
  4. Use interpolation for smooth visuals between updates

Physics Interpolation

[VISUAL: Animation showing choppy vs smooth movement]

Network updates happen at 20-30 Hz, but games render at 60+ FPS.

Without interpolation:

  • Position updates = choppy 20 FPS movement
  • Players teleport between positions

With interpolation:

  • Godot smoothly moves between received positions
  • Looks like 60 FPS even with 20 Hz updates

Enable it for remote players only:

if not is_multiplayer_authority():
	physics_interpolation_mode = Node.PHYSICS_INTERPOLATION_MODE_ON

Visibility Update Mode

visibility_update_mode = 1

Modes:

  • 0 = Always - Update even when off-screen (costs bandwidth)
  • 1 = Visible - Only update visible objects (optimization)

For a small multiplayer game, "Visible" saves bandwidth without noticeable impact.

What NOT to Synchronize

[VISUAL: Red X over examples]

❌ Local effects - Particle systems, sound effects (play them via RPC instead) ❌ Calculated values - Health bars, UI elements (update via RPC when value changes) ❌ Constant values - Player name (send once via RPC, not every frame)

Only synchronize: ✅ Frequently changing gameplay values (position, velocity, rotation) ✅ Visual states (animation, sprite flip, color) ✅ Values that need frame-accurate sync

Timing and Latency

[VISUAL: Timeline showing latency delays]

Player A Input:       T=0ms
Server Receives:      T=50ms (network latency)
Server Processes:     T=51ms (1ms server logic)
Server Broadcasts:    T=52ms
Other Clients Receive: T=102ms (50ms latency back)

Total time from input to seeing result: ~100ms

How to minimize perceived latency:

  • Instant local feedback (Player A sees jump immediately)
  • Interpolation (smooth remote player movement)
  • Prediction (clients predict physics between updates)
  • Server authority (prevents rollback/corrections from desync)

Network Optimization Techniques Used

[VISUAL: Checklist]

✅ State synchronization over input synchronization

  • Send "position is X" not "player pressed W"
  • Reduces bandwidth, easier to manage

✅ Authority-based processing

  • Only owner processes input
  • Only server runs game logic
  • Clear separation of responsibilities

✅ Reliable RPCs for state changes

  • Infection, scores, round transitions = reliable
  • Guarantees consistency across clients

✅ Automatic spawn/sync system

  • MultiplayerSpawner/Synchronizer reduce manual RPC calls
  • Less code = fewer bugs

✅ Late-join support

  • spawn = true sends initial values to new players
  • Player data sync system ensures everyone has full state

Chapter 8: Common Pitfalls and Debugging (35:00 - 38:00)

[VISUAL: Warning signs animation]

Pitfall #1: Calling Functions Before Peer is Set

# WRONG
var peer = ENetMultiplayerPeer.new()
my_function.rpc()  # FAILS - peer not set yet!
multiplayer.multiplayer_peer = peer

# RIGHT
var peer = ENetMultiplayerPeer.new()
multiplayer.multiplayer_peer = peer
my_function.rpc()  # Works!

Pitfall #2: Not Checking Server Authority

# WRONG - Runs on all clients!
@rpc("any_peer", "call_local", "reliable")
func give_player_points(amount: int):
	Global.score += amount

# RIGHT - Server validates
@rpc("any_peer", "call_remote", "reliable")
func give_player_points(amount: int):
	if not multiplayer.is_server():
		return

	# Server logic here
	update_score.rpc(Global.score + amount)

Pitfall #3: Synchronizing Too Much

# WRONG - Syncing 60 times per second!
extends CharacterBody2D

# MultiplayerSynchronizer syncing:
# - position
# - velocity
# - rotation
# - health
# - ammo
# - every single variable...

Solution: Only sync what remote players need to see!

Pitfall #4: Late Join Desync

# WRONG - New players don't know the score
func _ready():
	if multiplayer.is_server():
		score = 100  # Only server knows this!

# RIGHT - Sync to late joiners
@rpc("authority", "call_remote", "reliable")
func sync_game_state(current_score: int, round: int):
	score = current_score
	current_round = round

func _on_peer_connected(peer_id: int):
	if multiplayer.is_server():
		sync_game_state.rpc_id(peer_id, score, current_round)

RPC Patterns Cheat Sheet:

# Server tells everyone
@rpc("authority", "call_local", "reliable")
func update_state(data):
	pass
# Usage: update_state.rpc(data)

# Client asks server
@rpc("any_peer", "call_remote", "reliable")
func request_action():
	if multiplayer.is_server():
		# validate and respond
		pass
# Usage: request_action.rpc()

# Server tells specific client
@rpc("authority", "call_remote", "reliable")
func send_to_one(data):
	pass
# Usage: send_to_one.rpc_id(peer_id, data)

# Anyone tells everyone (cosmetic only!)
@rpc("any_peer", "call_local", "unreliable")
func play_effect():
	pass
# Usage: play_effect.rpc()

Glossary

  • Peer: A connected participant in the network (server or client)
  • Authority: The peer responsible for controlling an object
  • RPC: Remote Procedure Call - function executed across network
  • Reliable: Guaranteed delivery (may be slower)
  • Unreliable: Best-effort delivery (faster but may drop)
  • Spawn: Create an object across the network
  • Sync: Continuously update a property across the network
  • Latency: Time delay between sending and receiving data
  • Tick: Network update cycle (typically 30 Hz)
  • Interpolation: Smooth animation between network updates
